## 2. What post-research Does

Transforms short-form web content (tweets, Red Note posts, tech blog articles) into structured, reusable Markdown digests. Digests are saved to `digest/posts/`.

It uses a tiered fetch strategy that degrades gracefully:

```
Tier 0   Platform CLI          bird (Twitter/X) · redbook (Red Note)
         ↓ optional — skip if .env not configured or auth fails
Tier 1   Jina Reader           r.jina.ai/<url>
Tier 2   twitter-thread.com    Twitter/X threads only
Tier 3   defuddle.md           generic article fallback
Tier 4   markdown.new          last-resort converter
Tier 5   Search Synthesis      WebSearch — returns third-party commentary, NOT the original post
```

## 5. Credential Architecture

All credentials live in `.vibe/skills/post-research/.env` (gitignored). The `.env.example` template is committed with empty values:

```bash
# Twitter/X
TWITTER_AUTH_TOKEN=           # 40-char hex from Chrome DevTools → Cookies → x.com
TWITTER_CT0=                  # ~160-char alphanumeric from same location

# Red Note / Xiaohongshu
XHS_A1=                       # long alphanumeric from DevTools → Cookies → xiaohongshu.com
XHS_WEB_SESSION=              # starts with "040062..." from same location
```

**Resolution order** (all 4 scripts follow this):
1. `$TWITTER_AUTH_TOKEN` / `$TWITTER_CT0` / `$XHS_A1` / `$XHS_WEB_SESSION` env vars (already exported)
2. `.vibe/skills/post-research/.env` file (parsed line-by-line, `KEY=VALUE` format)
3. Browser extraction (macOS only — `bird`/`redbook` read Chrome cookies automatically)

**Exit code convention** for `*-check.sh`:
- `0` = authenticated → proceed with Tier 0
- `2` = not configured → **skip silently** to Tier 1 (NOT an error)
- `3` = configured but auth failed (tokens expired) → skip to Tier 1, warn user
- `1` = script-level error

**Backwards compatibility**: `XHS_COOKIE_STRING` (combined `a1=VAL; web_session=VAL`) is still accepted by the redbook scripts. The new individual-var approach (`XHS_A1` + `XHS_WEB_SESSION`) is preferred — scripts combine them automatically.

## 7. Key Design Decisions (Why Things Are The Way They Are)

1. **Why `.env` at skill root, not repo root?** Credentials are skill-specific (Twitter/Red Note tokens serve only `post-research`). Other skills shouldn't need to know about them. The `.gitignore` uses `**/.env` to catch any depth.

2. **Why separate vars (`XHS_A1` + `XHS_WEB_SESSION`) instead of combined `XHS_COOKIE_STRING`?** Easier for users to extract from DevTools (copy one cookie value at a time). Less error-prone than manually formatting `a1=VAL; web_session=VAL`. Scripts combine them internally.

3. **Why exit code 2 (not 1) for "not configured"?** Exit 1 conventionally means "error." Not having credentials is not an error — it's a valid, expected state. Exit 2 signals "skip this tier" without triggering error handling in the SKILL.md workflow.

4. **Why `bird` over `twitter-cli` (Python)?** `bird` is npm-based (consistent with the `redbook` approach), uses the same `@steipete/sweet-cookie` library, and has a rich CLI (`read`, `thread`, `search`, `bookmarks`). `twitter-cli` (Python, `uv tool install twitter-cli`) is a viable alternative with env var auth (`TWITTER_AUTH_TOKEN` + `TWITTER_CT0`) if `bird` breaks.

5. **Why Tier 0 is optional (not required)?** The user explicitly requested that the skill function fully without credentials. Missing credentials = degraded quality, not failure. The SKILL.md checklist says: "If `*-check.sh` exits non-zero... skip silently to Tier 1 — do NOT treat as an error."
